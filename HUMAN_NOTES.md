## Bugs

- [x] [BUG] Health scoring falsely inflates Code Quality and Dependency Health on greenfield (zero-source-file) projects, and the report provides no greenfield context. Two separate root causes: (1) In `_check_code_quality()` (`lib/health_checks.sh` ~L185), three inverse-penalty sub-scores are initialized to their **maximum** — `todo_score=20`, `magic_score=20`, `length_score=15` — and only reduced by scanning sample files. When `sample_files` is empty (no code yet), the `[[ -n "$sample_files" ]]` guard prevents scanning entirely, leaving all three at max. Result: 0+0+20+20+0+15 = 55/100 for Code Quality on a project with no code. Fix: change all three initializations to `0` and set them to their default-pass values (`20`/`20`/`15`) at the *top* of the `[[ -n "$sample_files" ]]` block, so they're only awarded when there's actual code to assess. (2) In `_check_dependency_health()` (`lib/health_checks_infra.sh` ~L73), `dep_ratio_score=25` is the unconditional default — the branch that would lower it only fires when `src_count > 0 && dep_count > 0`. A project with no manifest and no source files never enters that branch, so it receives a free 25 points. Fix: default `dep_ratio_score=0`; after the `manifest_score` for-loop (which already exists, ~L130 of `_check_dependency_health`), add `[[ "$manifest_score" -eq 0 ]] && dep_ratio_score=0` — this post-hoc clears the free-25 when no manifest was found. When `manifest_score > 0` but `src_count` is still 0 (manifest present, no source files yet), keeping `dep_ratio_score=25` is correct ("not over-dependent"). After fixes, an empty project with no manifest scores ~0 Code Quality and 0 Dependency Health, landing on White Belt ("Starting fresh") which is accurate. Additionally: in `_write_health_report()` (`lib/health.sh`), when `source_files == 0`, prepend a callout block ("⚠ Pre-code baseline — scores reflect project setup only, not code quality") before the dimension table. `source_files` is already in the `test_detail` JSON string (key `"source_files"`) — extract it with a `grep -oP` or simple `sed` on `$test_detail` before writing the report; no new function args needed. Test impact: `tests/test_health_scoring.sh` line 88 already uses `assert_range 0 35` which covers the post-fix composite (≈4–7) — do NOT change the range assertion. Instead add two targeted assertions after line 88: call `_check_code_quality "$EMPTY_DIR"` directly and assert the score field (pipe-field 2) is `0`; call `_check_dependency_health "$EMPTY_DIR"` and assert the score field is `0`. Also assert that `HEALTH_REPORT.md` contains the "Pre-code baseline" callout text.

Fix: add a fallback path at the end of `detect_languages()` — if the resulting `languages` string is empty and `${project_dir}/CLAUDE.md` exists, parse the CLAUDE.md for a tech stack section. The generated CLAUDE.md from `--plan` has no dedicated `## Tech Stack` heading — tech stack is a prose bullet under the `### 1. Project Identity` section (see `prompts/plan_generate.prompt.md` line 67: "Tech stack summary (languages, frameworks, key dependencies)"). Parse strategy: after the `### 1. Project Identity` line, scan until the next `###` heading and extract lines matching a known language name bullet. Specifically, lines whose first word after `- ` is a known language name (TypeScript, JavaScript, Python, Go, Rust, Java, Kotlin, Swift, Dart, Ruby, PHP, C#, Elixir, Haskell). A simple approach: extract the Project Identity block, then run `grep -iE "^-\s+(typescript|javascript|python|go|rust|java|kotlin|swift|dart|ruby|php|c#|elixir|haskell)"` and emit each match as `LANG|low|CLAUDE.md`. Because this is LLM-generated text, the language may appear mid-sentence in the tech stack summary bullet; if pure bullet extraction yields nothing, fall back to `grep -oiE` across the whole section. The `low` confidence already causes a `?` icon in `_display_detection_results()`, which is visually appropriate for planned-but-not-yet-written languages. No changes needed to callers — `detect_languages()` output format is unchanged. Add a test case to `tests/test_detect_languages.sh` (the existing language detection test file) covering: empty dir with a minimal CLAUDE.md containing a `### 1. Project Identity` section with a language bullet (e.g. `- TypeScript`) → expect non-empty language output with `|low|CLAUDE.md` confidence suffix.
