## Suspected Root Cause Areas

1. **Auto-advance loop initialization** — Milestone state must be created before stages run and preserved across loop iterations. State initialization at tekhton.sh:603-604 correctly calls `init_milestone_state()` with starting milestone extracted from task regex (tekhton.sh:598).

2. **Acceptance checking timing** — Must run after ALL stages complete (tester finishes) but before final checks and drift processing. Placement at tekhton.sh:684 and 742 is correct: after `_run_pipeline_stages()` returns.

3. **Disposition state machine** — Four transitions implemented: COMPLETE_AND_CONTINUE (loop again) → COMPLETE_AND_WAIT (stop, user hasn't finished) → INCOMPLETE_REWORK (stop, acceptance failed) → REPLAN_REQUIRED (reserved for M4). Lines 691, 708, 754 set appropriate dispositions.

4. **Task and context reset between iterations** — Task updated (tekhton.sh:717), START_AT reset to coder (line 721), reports archived (lines 724-730), timestamp/log renewed (lines 733-735). Necessary to avoid stale context.

5. **Report archiving between milestones** — Lines 724-730 archive reports from previous milestone with suffix `milestone${_CURRENT_MILESTONE}_${f}`. Keeps audit trail and prevents context corruption.

6. **User confirmation flow** — Conditional prompt at tekhton.sh:705 when AUTO_ADVANCE_CONFIRM=true. Accepts y/n, allows user to pause at any milestone boundary.

7. **Session limit enforcement** — Line 694 (`while should_auto_advance`) with `get_milestones_completed_this_session()` check at lib/milestones.sh:425. Caps loop at AUTO_ADVANCE_LIMIT (default: 3) to prevent runaway execution.

8. **Completion detection** — Two exit paths: find_next_milestone returns empty (line 696-699), or user declines confirmation (line 706-710). Both set COMPLETE_AND_WAIT and break cleanly.
