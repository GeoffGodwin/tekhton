## Architecture Decisions

### agent.sh sourced in --plan mode
The `--plan` block previously only sourced `common.sh`, `prompts.sh`, `plan.sh`,
and `plan_interview.sh`. The generation stage uses `run_agent()` from `agent.sh`
for batch-mode invocation, so `agent.sh` must now be sourced in the `--plan` block.
This is a minimal, additive change — no execution pipeline code is modified.

### PROJECT_NAME fallback
`run_agent()` uses `PROJECT_NAME` for temp file naming. In `--plan` mode,
`load_config()` is not called (config may not exist yet), so `PROJECT_NAME` is
derived from `basename "$PROJECT_DIR"` as a fallback. Uses `: "${PROJECT_NAME:=...}"`
so it won't override if already set.

### Generation model defaults
Added `PLAN_GENERATION_MODEL` (default: `sonnet` via `CLAUDE_PLAN_MODEL`) and
`PLAN_GENERATION_MAX_TURNS` (default: 30) following the same pattern as
interview config. 30 turns is sufficient for a single-file generation task.

## Architecture Change Proposals

### agent.sh now sourced in --plan early-exit block

- **Current constraint**: The `--plan` block sources only `common.sh`, `prompts.sh`,
  `plan.sh`, and `plan_interview.sh` — keeping dependencies minimal since config
  may not exist yet.
- **What triggered this**: The generation stage uses `run_agent()` from `agent.sh`
  for batch-mode agent invocation. Without it, generation cannot run.
- **Proposed change**: Source `agent.sh` in the `--plan` block alongside the other
  libraries. `agent.sh` has no config dependencies — it initializes its own globals
  with safe defaults (`: "${TOTAL_TURNS:=0}"` etc.).
- **Backward compatible**: Yes — no existing behavior changes.
- **ARCHITECTURE.md update needed**: Yes — Layer 1 section should note that
  `--plan` now sources `agent.sh` in addition to existing libraries.
