## Bugs Found and Fixed

The Scout agent identified 4 bugs across 2 files. All have been fixed and verified.

### Bug 1: Single-Bracket Conditional (FIXED)
- **Location:** `tests/test_agent_fifo_invocation.sh` line 260
- **Issue:** Used single-bracket conditional `[ "$FAIL" -ne 0 ]` instead of double-bracket form
- **Impact:** Violates Tekhton bash style requirement for conditionals; inconsistent with project patterns
- **Fix Applied:** Changed to `[[ "$FAIL" -ne 0 ]]` (double-bracket conditional)
- **Verified:** ✓ Test file follows bash style conventions

### Bug 2: Misleading PATH Assignments (FIXED)
- **Location:** `tests/test_agent_fifo_invocation.sh` lines 125, 155, 185, 217, 238, 241
- **Issue:** Used pattern `PATH="${MOCK_BIN}:${PATH}" run_agent ...` at 6 call sites, which looks temporary but is permanent in bash. When `run_agent` is a shell function (not an external command), the `VAR=value func` syntax does NOT create a temporary environment — it permanently modifies PATH and accumulates duplicate entries.
- **Impact:** PATH variable polluted on each call, creating maintenance issues and misleading code readers
- **Fix Applied:** Moved to single `export PATH="${MOCK_BIN}:${PATH}"` at top of file (line 32), before any run_agent calls. All 6 call sites cleaned up.
- **Verified:** ✓ PATH is set once at file initialization, tests run cleanly

### Bug 3: TMPDIR Variable Shadowing (FIXED)
- **Location:** `tests/test_agent_fifo_invocation.sh` line 17
- **Issue:** Used `TMPDIR=$(mktemp -d)` which shadows the system `$TMPDIR` environment variable used by `mktemp` and other system tools
- **Impact:** Potential for subtle failures if nested mktemp calls or other tools rely on the system TMPDIR variable
- **Fix Applied:** Renamed to `TEST_TMP=$(mktemp -d)` throughout the file (lines 17, 18, 28, 36, 40)
- **Verified:** ✓ System TMPDIR environment variable is no longer shadowed

### Bug 4: Confusing Timeout Message (FIXED)
- **Location:** `lib/agent.sh` line 361
- **Issue:** When `AGENT_TIMEOUT=0` (outer timeout disabled), the agent activity timeout could trigger, but the null-run error message always reported `"timed out after ${_timeout}s"` — which is `0s`, creating confusion. The message should distinguish between activity timeout and outer timeout.
- **Impact:** Misleading error messages make debugging timeout issues difficult; doesn't reflect which timeout actually triggered
- **Fix Applied:** Added conditional logic at `lib/agent.sh:361-365`:
  ```bash
  if [ "$_was_activity_timeout" = true ]; then
      warn "[$label] NULL RUN DETECTED — agent activity-timed out after ${_activity_timeout}s of silence."
  else
      warn "[$label] NULL RUN DETECTED — agent timed out after ${_timeout}s."
  fi
  ```
- **Verified:** ✓ Test Phase 6 (activity timeout) verifies message contains "ACTIVITY TIMEOUT"
