## Suspected Root Cause Areas

1. **Parser output format mismatch** — If `_extract_template_sections()` is extended to 4 fields but interview code still reads 3 fields, parsing will fail silently (field 4 goes to guidance)

2. **Phase marker syntax inconsistency** — If templates use inconsistent marker formats (e.g., `<!-- PHASE:1 -->` vs `<!-- PHASE 1 -->` vs `<!-- Phase 1 -->`), parser won't recognize them. Must standardize on `<!-- PHASE:N -->`

3. **Context display implementation** — Showing Phase 1 answers before Phase 2 questions requires:
   - Building answer summary from Phase 1 data structure
   - Passing it to `_read_section_answer()` or wrapping it with additional guidance
   - If not carefully handled, Phase 1 content could be duplicated or accidentally merged into Phase 2 answers

4. **Resume state granularity** — If plan_state.sh only tracks interview/completeness/generation (coarse) but not which phase was interrupted, resume from Phase 2 mid-interview will re-ask Phase 1 questions or lose Phase 1 answers

5. **Prompt synthesis timing** — Currently, `run_plan_interview()` collects all answers then calls synthesis once. With three phases, decisions needed on:
   - Collect all 3 phases, then one synthesis pass (simpler, but context could be cold for Phase 3)
   - Synthesize after each phase (more complex state management, but keeps context warm)
   - Milestone 2 spec suggests one synthesis pass with all answers available (simpler approach)
