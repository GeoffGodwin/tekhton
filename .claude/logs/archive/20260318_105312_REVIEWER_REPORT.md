## Non-Blocking Notes
- `handle_null_run_split()` uses `git diff --stat HEAD` to detect substantive work, but the condition that activates it checks `git diff --quiet` (unstaged) OR `git diff --cached --quiet` (staged). `git diff HEAD` captures both staged and unstaged changes correctly, so behavior is correct — but a comment explaining that `git diff HEAD` is intentional (not `git diff`) would prevent future confusion.
- The recursive call to `run_stage_coder` in the null-run auto-split path creates nested call frames up to `MILESTONE_MAX_SPLIT_DEPTH` deep. With the default of 3 this is safe, but a log line indicating the recursion depth would aid debugging (e.g., "Auto-split retry (depth 2/3)").
- The `get_milestone_title` call immediately after `split_milestone()` returns success assumes the splitting agent produces headings in exactly the format `get_milestone_title` expects. If the split output uses a slightly different heading format, the title falls back to an empty string silently (the `|| true` guard). Acceptable since the task still proceeds; worth a comment.
- `MILESTONE_SPLIT_MAX_TURNS` is clamped at 50 in config.sh, but `MILESTONE_SPLIT_MAX_TURNS=15` is the configured default. The hard cap of 50 is reasonable but undocumented in `pipeline.conf.example` — the example should note the cap so operators know the ceiling.
