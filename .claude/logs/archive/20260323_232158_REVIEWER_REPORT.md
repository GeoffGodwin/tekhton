## Non-Blocking Notes
- `lib/diagnose.sh` is 626 lines, well over the 300-line soft ceiling. Consider extracting `_collapse_cause_chain` / `_detect_recurring_failures` / `_collect_agent_log_tails` into a `lib/diagnose_helpers.sh` in the next cleanup pass.
- `_detect_recurring_failures()` is called twice: once from `_read_diagnostic_context()` (line 114 of diagnose.sh) where it is a complete no-op because `DIAG_CLASSIFICATION` is empty at that point, and once from `classify_failure_diag()` after classification where it actually executes. The first call is dead code — can be removed without behavior change.
- `_hook_failure_context` (hook 13) runs after `_hook_causal_log_finalize` (hook 4), which archives the causal log. If `archive_causal_log()` moves (rather than copies) the live log file, `_read_diagnostic_context()` inside `_hook_failure_context` will silently find no causal events and produce a less accurate inline classification. Worth adding a comment in `_hook_failure_context` noting the post-archive execution context.
- `_rule_review_loop`'s causal-log fallback path (triggered when `exit_stage != "review"`) counts all reviewer verdict events without filtering for CHANGES_REQUIRED. Three APPROVED verdicts in a successful run would trigger this path if the pipeline later failed for another reason. Low risk in practice (advisory classifier only), but worth adding a verdict type filter.
- When `_DIAG_REVIEW_CYCLES` is not populated from RUN_SUMMARY.json, `_rule_review_loop` defaults to 3 in its suggestion text ("Reviewer rejected the code 3 times") — a hard-coded fallback rather than a derived value. Minor cosmetic issue.

## Coverage Gaps
- No test for `print_run_report()` in lib/report.sh. The summary rendering path (header, outcome coloring, action items, --diagnose hint) would benefit from at least a smoke test to catch regressions.

## Drift Observations
- `diagnose_rules.sh:299` — `# shellcheck disable=SC2034` is placed above the `_rule_unknown()` function definition line. SC2034 disables apply to the immediately following statement, not the function body, so any suppressed assignment inside the function is not actually covered. Shellcheck reports clean, so benign — but the comment placement may confuse future readers.
- `lib/state.sh:112-119` — `clear_pipeline_state()` uses `[ -f ]` (POSIX single brackets) for both its original check and the new M17 addition. Project standard is `[[ ]]`. Pre-existing inconsistency in the function; M17 added code matches existing local style rather than the project standard.
