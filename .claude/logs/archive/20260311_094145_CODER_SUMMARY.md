## Architecture Change Proposals

### New file: `lib/plan_state.sh`
- **Current constraint**: Architecture lists planning code in `lib/plan.sh`, `stages/plan_interview.sh`, `stages/plan_generate.sh`
- **What triggered this**: `lib/plan.sh` exceeded the 300-line limit (440 lines) with state persistence functions inline
- **Proposed change**: Extracted state persistence to `lib/plan_state.sh`, following the same pattern as `lib/plan_completeness.sh` extraction. The file is sourced in tekhton.sh's `--plan` block.
- **Backward compatible**: Yes â€” no existing code is affected
- **ARCHITECTURE.md update needed**: Yes â€” add `lib/plan_state.sh` to the System Map and File Ownership sections. Update the `--plan` source list in the Layer 1 description.

### Planning config loaded before config.sh
- **Current constraint**: Architecture implies planning config defaults are hard-coded in `lib/plan.sh`
- **What triggered this**: Milestone 6 requires pipeline.conf integration for planning config keys
- **Proposed change**: `load_plan_config()` in `lib/plan.sh` optionally sources `pipeline.conf` at load time (before the execution pipeline's `load_config()` runs). This is safe because `load_config()` validates required keys separately. Planning keys are all optional with defaults.
- **Backward compatible**: Yes â€” if pipeline.conf doesn't exist or has no planning keys, defaults apply unchanged
- **ARCHITECTURE.md update needed**: Yes â€” note in Layer 1 that `--plan` optionally reads pipeline.conf for planning config keys
