## What Was Implemented

Milestone 3: Completeness Check + Follow-Up — full implementation:

- **`check_design_completeness()`** in `lib/plan.sh` — deterministic grep/awk-based
  section validation that cross-references DESIGN.md against REQUIRED markers in the
  selected template. Detects: empty sections, guidance comments still present,
  placeholder-only content (TBD, TODO, TK, N/A, TBA).

- **`run_plan_completeness_loop()`** in `lib/plan.sh` — orchestrates the completeness
  check + follow-up cycle. Reports incomplete sections to the user, offers follow-up
  interview or skip. Caps at 3 follow-up passes to prevent infinite loops.

- **`run_plan_followup_interview()`** in `stages/plan_interview.sh` — launches a
  focused follow-up interview in conversational mode, targeting only incomplete
  sections. Preserves all existing DESIGN.md content.

- **`prompts/plan_interview_followup.prompt.md`** — follow-up agent prompt that
  receives the current DESIGN.md content and list of incomplete sections.

- **`tests/test_plan_completeness.sh`** — 20 test cases covering: required section
  extraction, incomplete detection (empty, whitespace, guidance comments, placeholders,
  real content, mixed content), full completeness check (complete doc, empty section,
  missing section, guidance-only, TBD placeholder), and determinism verification.

- **Wiring** — `run_plan()` now calls `run_plan_completeness_loop()` after the
  interview step (Step 3), before the future generation step.

## Architecture Decisions

- **Completeness loop lives in `lib/plan.sh`**, not in `stages/plan_interview.sh`.
  The interview stage only handles launching claude sessions. The orchestration
  logic (check → prompt user → decide next action) belongs in the library layer,
  consistent with how `run_plan()` orchestrates all planning steps.

- **Max 3 follow-up passes** as a safety cap. This prevents infinite loops if
  the LLM consistently fails to fill sections properly. After 3 passes, the user
  can edit DESIGN.md manually.

- **`PLAN_INCOMPLETE_SECTIONS` as a global variable** (newline-separated list)
  is the communication channel between `check_design_completeness()` and
  `run_plan_followup_interview()`. This follows the existing pattern of using
  shell globals for inter-function state (e.g., `CODER_STATUS`, `CODER_REMAINING`
  in `lib/gates.sh`).
