# Scout Report — Milestone 13.1: Retry Infrastructure

## Relevant Files

- `lib/config.sh` — Configuration loading with defaults and clamping mechanism. Already has pattern for adding new config keys with defaults via `: "${VAR:=value}"` and clamping via `_clamp_config_value()`. Will add 4 new config keys: `MAX_TRANSIENT_RETRIES`, `TRANSIENT_RETRY_BASE_DELAY`, `TRANSIENT_RETRY_MAX_DELAY`, `TRANSIENT_RETRY_ENABLED` with corresponding clamp calls.

- `lib/common.sh` — Logging and formatted output utilities. Already contains `_is_utf8_terminal()` and `report_error()` which demonstrates the Unicode/ASCII box-drawing pattern that `report_retry()` must follow. Will add new `report_retry()` function following the same structure.

- `lib/agent_monitor.sh` — Agent process monitoring with FIFO, ring buffer for last output, and real-time API error detection flags (`_API_ERROR_DETECTED`, `_API_ERROR_TYPE`). Contains process ID tracking via `_TEKHTON_AGENT_PID` and cleanup routines. Will add `_reset_monitoring_state()` to clean up monitoring artifacts between retry attempts.

- `templates/pipeline.conf.example` — Configuration template documentation. Will add new section `# --- Transient error retry` with explanatory comments for the 4 new config keys.

## Key Symbols

- `_is_utf8_terminal()` in `lib/common.sh` — Function returning 0 if terminal supports UTF-8, used for Unicode box-drawing character selection
- `report_error(category, subcategory, transient, message, recovery)` in `lib/common.sh` — Existing structured error reporting function that uses box-drawing for stderr output; `report_retry()` will follow identical pattern
- `_clamp_config_value()` in `lib/config.sh` — Safety function preventing config value overruns; used to cap turn limits, timeout values, retry counts
- `_TEKHTON_AGENT_PID` in `lib/agent_monitor.sh` — Background claude process ID, needed for cleanup
- `_API_ERROR_DETECTED`, `_API_ERROR_TYPE` in `lib/agent_monitor.sh` — Monitoring flags set by FIFO reader for API error detection; will be reset by `_reset_monitoring_state()`
- `_invoke_and_monitor()` in `lib/agent_monitor.sh` — Core agent invocation function managing FIFO, ring buffer, activity timeout, and exit code tracking

## Suspected Root Cause Areas

- Config defaults and clamps must follow established patterns in `load_config()` to maintain consistency with existing 20+ config keys already there
- `report_retry()` must use same Unicode/ASCII logic as `report_error()` for consistency; pattern is established and proven
- `_reset_monitoring_state()` must safely handle missing FIFO/temp files and background process cleanup without breaking existing monitoring flow
- Temporary files created during monitoring (`agent_stderr.txt`, `agent_last_output.txt`, `agent_api_error.txt`, `agent_exit`, `agent_last_turns`) must all be cleaned up by `_reset_monitoring_state()` between retry attempts
