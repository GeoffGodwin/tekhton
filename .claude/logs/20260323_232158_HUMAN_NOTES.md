## Bugs
- [~] [BUG] Fix the Watchtower dashboard deployment so static UI files are always present in .claude/dashboard/.
Bug: The dashboard's static files (index.html, style.css, app.js) are never copied from 
templates/watchtower/ into .claude/dashboard/. The root cause is in tekhton.sh around line 1003: 
init_dashboard() is only called when .claude/dashboard/ does not exist. But the data/ subdirectory 
gets created on the first run, so all subsequent runs skip init_dashboard entirely. If the template 
files were added after the dashboard directory was first created (which happened — M14 added 
templates/watchtower/ after M13 created the dashboard data dir), the static files never get copied.
The data/*.js files are fine — they get updated by emit_dashboard_* calls every run. It's only 
the static UI scaffolding (index.html, style.css, app.js) that's missing.
Fix: In tekhton.sh, ensure the static UI files are copied on every startup when the dashboard 
is enabled, not just on first creation. The simplest approach: after the existing init_dashboard 
guard, always call _copy_static_files (or an equivalent) to sync templates/watchtower/ into 
.claude/dashboard/. This also ensures template updates propagate to deployed dashboards.
Additionally, while you have the dashboard front-end open: the footer of index.html says 
"no server required" but also has a Safari file:// workaround note. Audit app.js for any 
fetch(), dynamic import(), or other patterns that break under file:// protocol and fix them. 
The data/*.js files use a window global pattern (e.g. window.TK_RUN_STATE = {...}) which works 
under file:// — make sure app.js only reads those globals and doesn't try to fetch the data 
files separately.
Constraints: Dashboard must work by opening .claude/dashboard/index.html directly in a browser (no server). All existing tests must still pass (bash tests/run_tests.sh).
