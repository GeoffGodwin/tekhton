## Suspected Root Cause Areas

- **State file parsing** — `_read_diagnostic_context()` must safely parse CAUSAL_LOG.jsonl (JSONL format with line-based events), PIPELINE_STATE.md (markdown key-value pairs), RUN_SUMMARY.json (JSON), MANIFEST.cfg (INI-like), and error files. Defensive parsing required to handle missing/malformed files gracefully.

- **Rule priority and causal chain influence** — BUILD_FAILURE rule must be checked before STUCK_LOOP to provide build-specific advice even when stuck loops are caused by builds. Causal chain often reveals this naturally, but rule order is the safety net.

- **Forward compatibility pattern** — Rules for future stages (security, intake, quota) must check for file existence (e.g., `[[ -f "$SECURITY_REPORT_FILE" ]]`) before matching. Missing file = rule returns 1 (no match). This allows --diagnose to work without code changes when new stages land.

- **Atomic LAST_FAILURE_CONTEXT.json writes** — Must use tmpfile+mv pattern to ensure JSON file survives crashes. Difficult to test; watch for race conditions or partial writes.

- **Terminal summary verbosity** — Causal chains can be verbose (20+ events). Terminal summary must show ≤5 links; full chain goes in DIAGNOSIS.md. Requires careful event collapsing logic (e.g., "3 rework_cycles" instead of listing each).

- **Integration with crash handler** — Enhanced crash handler at tekhton.sh:55-99 must run fast (no agent calls) and provide context-appropriate advice. Requires quick checks: claude CLI auth, BUILD_ERRORS.md existence, PIPELINE_STATE.md presence, transient error patterns in agent log tail.

- **Dashboard.js generation** — `emit_dashboard_diagnosis()` must produce valid JavaScript object assignment (`window.TK_DIAGNOSIS = {...}`) consumable by Watchtower. Schema and timing of writes must be coordinated with finalize sequence.

- **Resume prompt enhancement** — LAST_FAILURE_CONTEXT.json must be written early (before pipeline can crash) but after failure is detected. Timing in finalize sequence is critical.
