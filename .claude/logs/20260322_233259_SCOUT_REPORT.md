## Suspected Root Cause Areas

- **Personalization blending logic** — `_rank_files()` in repo_map.py needs to: (1) load history file if `--history-file` provided, (2) extract keywords from current task, (3) find similar past tasks using bag-of-words overlap, (4) build personalization vector mixing keyword relevance (0.6), historical file relevance from similar tasks (0.3), and git recency (0.1), (5) bias PageRank with this vector.
- **Cache statistics** — TagCache needs mutable state for hit/miss/time counters; must be thread-safe (atomic writes) and persisted in JSON alongside tag entries.
- **Cache versioning** — TagCache format may change between Tekhton versions (e.g., adding stats fields). Must detect format mismatch on load and trigger rebuild rather than crash.
- **History file pruning** — Task history grows unbounded; must implement append-safe pruning that keeps only the most recent N records (configurable via REPO_MAP_HISTORY_MAX_RECORDS), creates new file, atomically replaces old.
- **Integration point timing** — `record_task_file_association()` must be called AFTER coder stage succeeds but BEFORE metrics are recorded, ensuring history is logged for that run's metrics.
- **Warm cache progress** — For large projects (10k+ files), warm cache may take 30-60 seconds. Must show periodic status to avoid timeout perception.

## Notes

- Task similarity is keyword-based (bag-of-words), not semantic — avoids cost and complexity of embeddings for marginal gain.
- Git recency signal requires a git repo; for non-git projects, redistribute weights (0.7 keyword, 0.3 history).
- History file lives in `.claude/` which should be gitignored. Add warning to setup output about sensitive task descriptions.
- JSONL format is append-only by design — never read-modify-write. Pruning creates new file and atomically replaces.
- Config keys (REPO_MAP_HISTORY_ENABLED, REPO_MAP_HISTORY_MAX_RECORDS) already in lib/config_defaults.sh; just need documentation in templates/pipeline.conf.example.
