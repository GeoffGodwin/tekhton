## Suspected Root Cause Areas

1. **Misleading success message (lib/plan.sh:224)** — `success "Files written to ${PROJECT_DIR}:"` is factually incorrect. Files were already written by `run_plan_generate()` in Step 4; the review step is just approval/editing. Message should reflect that files are ready/confirmed, not newly written.

2. **Unguarded editor invocation (lib/plan.sh:232)** — `${EDITOR:-nano} "$claude_file"` has no error handling. If the editor exits non-zero (e.g., user kills terminal), `set -euo pipefail` causes `run_plan_review()` to abort unexpectedly. Needs `|| true` or explicit error handling.

3. **Double file read in _display_milestone_summary (lib/plan.sh:149-152)** — Calls `_extract_project_name()` (which greps CLAUDE.md at line 129) and `_extract_milestones()` (which greps CLAUDE.md at line 140). Two separate file reads for information that could be extracted in a single pass. Not a correctness bug, but suboptimal for performance (though negligible in practice).

## Affected Code Sections

### Issue 1: Line 224 in lib/plan.sh
```bash

    success "Files written to ${PROJECT_DIR}:"  # ← MISLEADING
    log "  DESIGN.md"
    log "  CLAUDE.md"

```
**Fix**: Change message to "Files are ready at ${PROJECT_DIR}:" or "Files confirmed."

### Issue 2: Line 232 in lib/plan.sh
```bash
e|E)
    log "Opening CLAUDE.md in editor..."
    ${EDITOR:-nano} "$claude_file"  # ← NO ERROR GUARD
    log "Editor closed. Refreshing milestone summary..."
    ;;
```
**Fix**: Add `|| warn "Editor exited with non-zero status"` or similar after editor invocation.

### Issue 3: Lines 149-152 in lib/plan.sh
```bash

    ...
}
```
**Fix**: Refactor to read file once and extract both values, or cache the file content.
