### SF-3: Fix `trendArrow()` load-bearing assumption comment (`templates/watchtower/app.js:713-716`)
`trendArrow()` silently assumes `runs` is sorted newest-first and uses `slice(0,10)` and
`slice(10,20)` hardcoded. Neither the sort order nor the magic numbers are explained in a comment.

**Action:** Add a comment above the `slice` calls:
```js
// Assumes runs[] is sorted newest-first (as emitted by dashboard_parsers.sh).
// recent = runs[0..9], prior = runs[10..19].
```
This documents the load-bearing invariant without requiring a code change or test.

### SF-4: Normalize `dashboard.sh` sourcing pattern for `dashboard_parsers.sh` (`lib/dashboard.sh:24`)
`dashboard.sh` sources `dashboard_parsers.sh` via a `BASH_SOURCE`-relative path:
```bash
source "$(dirname "${BASH_SOURCE[0]}")/dashboard_parsers.sh"
```
Every other companion-file source in the codebase uses `${TEKHTON_HOME}/lib/`.

**Action:** Replace with:
```bash
source "${TEKHTON_HOME}/lib/dashboard_parsers.sh"
```
This matches the established pattern in `agent.sh`, `indexer.sh`, and all other library
files. Verify that `TEKHTON_HOME` is always set before `dashboard.sh` is sourced (it is —
it is resolved at the very top of `tekhton.sh` before any library is sourced).

## Simplification

Route to sr coder.

### SX-1: Extract shared dimension-check + composite-calculation block in `lib/health.sh`
`assess_project_health()` (lines ~55-137) and `reassess_project_health()` (lines ~144-250)
each contain a verbatim copy of:
1. Weight variable declarations (`w_tests`, `w_quality`, `w_deps`, `w_docs`, `w_hygiene`)
2. Five `_check_*` invocations with result + detail parsing via `cut`
3. The composite arithmetic and clamp to [0, 100]

This is the largest deliberate duplication in M15 and is now load-bearing as a maintenance
hazard: a weight formula change must be applied in two places.

**Action:** Extract the shared block into a private helper, e.g.:
```bash
# _run_health_dimensions PROJECT_DIR
# Outputs: composite score, belt, subtitle, five scores, five detail blobs
# (as a structured string or multiple output variables via nameref / a temp file)
_run_health_dimensions() { ... }
```
Both `assess_project_health` and `reassess_project_health` call `_run_health_dimensions`
and then diverge only in what they do with the results (baseline write vs. delta
computation). Scope is bounded to `lib/health.sh` only; no callers change signatures.

### SX-2: Replace `grep -c '.'` with `wc -l` for `sample_count` (`lib/health_checks.sh:279`)
```bash
sample_count=$(echo "$sample_files" | grep -c '.' || true)
```
This spawns two processes (`echo` + `grep`) to count non-empty lines. The idiomatic
single-process alternative:
```bash
sample_count=$(echo "$sample_files" | wc -l)
```
or, if empty-string handling is needed:
```bash
[[ -z "$sample_files" ]] && sample_count=0 || sample_count=$(echo "$sample_files" | wc -l)
```
**Action:** Replace the `grep -c '.'` expression with `wc -l` with an appropriate empty
guard. Verify that the surrounding logic (line 280: `[[ "$sample_count" -gt 0 ]]`) still
holds when `sample_files` is empty.

## Design Doc Observations

Route to human via `HUMAN_ACTION_REQUIRED.md`.

### DDO-1: `_json_escape` / output utility locality (causality.sh vs. dashboard_parsers.sh)
`_json_escape` lives in `lib/causality.sh` while `_to_js_string` and `_write_js_file`
live in `lib/dashboard_parsers.sh`. As the dashboard grows, contributors lack a clear
rule for where to place new JSON/JS output helpers.

**Human decision needed:** Should a dedicated `lib/output_helpers.sh` (or
`lib/json_helpers.sh`) be created to consolidate all low-level serialization utilities?
This would require moving `_json_escape` out of `causality.sh` and updating all callers —
a non-trivial refactor. Alternatively, document the current split explicitly in
`ARCHITECTURE.md` (causality.sh owns JSON primitives; dashboard_parsers.sh owns JS-file
output) so future contributors have a decision rule without a code change.

Record the decision in `ARCHITECTURE.md` under the causality/dashboard module
descriptions.
