## Non-Blocking Notes
- `lib/mcp.sh` header comment (line 7) says "Used by agent_monitor.sh to add --mcp-config flag" but the actual consumer is `agent.sh` (line 132). Update the docstring to `agent.sh`.
- `_SERENA_DIR` and `_SERENA_PYTHON` are never initialized at module level in `mcp.sh`, unlike the other module-state variables (`_MCP_SERVER_PID=""`, `_MCP_SERVER_RUNNING=false`, `_MCP_CONFIG_PATH=""`). Access is properly guarded so there's no functional bug, but the inconsistency could confuse future maintainers. Add `_SERENA_DIR=""` and `_SERENA_PYTHON=""` to the module state block.
- `check_mcp_health()` (line 49) contains a dead-code branch — `_MCP_SERVER_PID` is never set because Claude CLI owns the process lifecycle. The `kill -0` check will never execute. Either remove the branch or add a comment noting it's reserved for future per-agent server instances.
- Several private functions in `mcp.sh` (`_resolve_serena_paths`, `_resolve_mcp_config`) and most of `setup_serena.sh` use single-bracket `[ ]` for file/directory tests where the project convention (and reviewer checklist) calls for `[[ ]]`. Shellcheck does not flag this, and the existing `tekhton.sh` itself mixes both styles, so this is a low-priority consistency note rather than a hard blocker.
- `--with-lsp` in the usage block (tekhton.sh line 36/410) has inconsistent indentation vs `--setup-indexer` above it — extra spaces make it visually misaligned.
- ARCHITECTURE.md is out of date: `lib/mcp.sh` is now in Layer 3 and the MCP lifecycle should be described in the data flow section. The ACP already flags this — it should be addressed soon (next human-authored update or dedicated cleanup pass).
- No `tests/test_mcp.sh` was added. Other modules at this depth (dag, indexer) have associated test files. Consider adding at minimum: config generation roundtrip, `start_mcp_server()` when `SERENA_ENABLED=false`, and `check_mcp_health()` state transitions.

## ACP Verdicts
- ACP: Claude CLI MCP Server Management — **ACCEPT** — Delegating server process lifecycle to Claude CLI via `--mcp-config` is architecturally correct: avoids orphan process risk, is compatible with future parallel-agent execution (each session gets its own lifecycle), and eliminates the startup-timeout concern. The `start_mcp_server()` / `stop_mcp_server()` interface is preserved for state tracking and future extensibility. The ARCHITECTURE.md update noted in the ACP should be applied.

## Drift Observations
- `lib/mcp.sh:143` — `_cli_supports_mcp_config()` shells out to `claude --help | grep` on every `start_mcp_server()` call. If Claude CLI is slow to start or the help text grows, this adds latency on every pipeline run where `SERENA_ENABLED=true`. Consider caching the result in a module-level variable (`_CLI_SUPPORTS_MCP_CONFIG=""`) after the first check.
- `tools/setup_serena.sh` language-server detection enumerates by binary name (`pyright`, `typescript-language-server`, etc.) but does not validate that Serena actually knows how to configure those servers. If Serena's config schema differs from the detected server name, the generated `serena_mcp_config.json` may be syntactically valid but functionally broken. Worth adding a note in the setup summary output.
