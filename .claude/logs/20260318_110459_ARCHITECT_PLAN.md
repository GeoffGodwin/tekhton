## Simplification

**Item P-1: Extract `_switch_to_sub_milestone()` helper in `stages/coder.sh`**

The pattern of computing `_first_sub` and calling `get_milestone_title` / `init_milestone_state`
after a milestone split appears in three independent call sites in `stages/coder.sh`:

- **Pre-flight split path** (lines 84–93): computes `_first_sub`, calls `get_milestone_title`,
  sets `_CURRENT_MILESTONE`, updates `TASK`, calls `init_milestone_state`
- **Null-run auto-split path** (lines 329–341): same 8-line sequence
- **Turn-limit minimal-output path** (lines 465–472): same 8-line sequence

The drift log flagged this when two sites existed and noted "not worth abstracting now but
worth noting if a third call site appears." There are now three sites. The duplication
threshold has been crossed.

Action: Extract a shell function `_switch_to_sub_milestone(milestone_num, claude_md)` that
encapsulates the repeated sequence:
1. Compute `_first_sub="${milestone_num}.1"`
2. Call `get_milestone_title "$_first_sub" "$claude_md" 2>/dev/null || true`
3. Set `_CURRENT_MILESTONE="$_first_sub"` (callers must handle this via a local variable
   or by referencing the outer scope — use `declare -g` or pass back via a named-ref
   variable to avoid subshell issues)
4. Set `TASK="Implement Milestone ${_first_sub}: ${_first_title}"`
5. Call `log "Task updated..."`
6. Call `init_milestone_state "$_first_sub" "$(get_milestone_count "$claude_md")"`

The function must NOT call `run_stage_coder` — the recursion / return behavior differs
between the three call sites and should remain at call-site level. Only the 6 shared lines
above should be extracted.

Place the helper near the top of `stages/coder.sh` with the other local helpers, before
`run_stage_coder()`. Callers replace their repeated block with a single call to
`_switch_to_sub_milestone "$_CURRENT_MILESTONE" "CLAUDE.md"`.

Run `shellcheck` and `bash -n` on `stages/coder.sh` after the edit. All existing tests
in `tests/test_milestone_split.sh` must continue to pass.

This resolves drift observation: "`stages/coder.sh` lines 190–196... pattern is repeated in
two auto-split paths... Minor duplication — not worth abstracting now but worth noting if a
third call site appears."

## Out of Scope

**`tekhton.sh:1076` — `%%` strip on multi-line commit message**

The drift observation notes that `${COMMIT_MSG%%$'\n'*}` "may behave differently across
bash versions — worth watching if users report truncated suggestions." This is speculative:
(1) CLAUDE.md requires bash 4+ and the `%%` parameter expansion on `$'\n'` is stable across
bash 4+, (2) the strip is intentional — it displays a single-line command suggestion when
the user skips the commit prompt, (3) no users have reported truncated suggestions. No
actionable fix exists that would not change intentional behavior. Watch-and-wait is the
correct disposition. Out of scope for this audit cycle.
