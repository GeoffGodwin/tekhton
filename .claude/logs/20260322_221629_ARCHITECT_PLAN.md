## Staleness Fixes

**SF-1** — `tools/setup_serena.sh` summary output lacks a caveat about language server name validation.

The script detects language servers by binary name (`pyright`, `typescript-language-server`, etc.) and writes those names into `serena_mcp_config.json` via template substitution. The summary printed at the end of the script (lines 243–253) does not inform the user that Serena's config schema may not recognize these server names exactly, leading to a silently-broken config. Add a warning line to the summary block (after the "Language servers:" line) noting that detected names are based on PATH binaries and the generated config should be verified against Serena's supported server identifiers before use.

- **File**: `tools/setup_serena.sh`, lines 241–266 (summary + gitignore section)
- **Change**: Add a `_warn` line after line 247 if `$detected_servers != "none"` explaining that server names come from PATH binary detection and may need adjustment to match Serena's config schema.
- **Scope**: 3–4 lines; no logic change.

## Simplification

**SX-1** — `lib/mcp.sh:_cli_supports_mcp_config()` forks a `claude --help | grep` subprocess on every `start_mcp_server()` call.

Since `start_mcp_server()` is called during pipeline startup and the Claude CLI version does not change mid-run, the result is stable for the lifetime of the process. Cache the result in a module-level variable so the subprocess fork happens at most once per pipeline run.

- **File**: `lib/mcp.sh`, lines 138–147 and 162 (function definition + call site)
- **Change**: Declare `_CLI_MCP_CONFIG_SUPPORTED=""` at module level. Inside `_cli_supports_mcp_config()`, if the variable is already set, return based on its value; otherwise run the check, store `"1"` or `"0"`, and return accordingly.
- **Scope**: ~8 lines added, no behavior change when called once. Eliminates repeated subprocess cost for the common case where SERENA_ENABLED remains true across continuation attempts.

**SX-2** — `tools/repo_map.py:_filter_by_extension()` imports the private dict `_EXT_TO_LANG` directly from `tree_sitter_languages.py`.

The import `from tree_sitter_languages import _EXT_TO_LANG` (line 113) creates hidden coupling to an internal data structure. If `_EXT_TO_LANG` is ever renamed, restructured, or replaced with a class attribute, `repo_map.py` fails with an `ImportError` rather than a clear `AttributeError` at the call site, and the coupling is invisible to callers of `tree_sitter_languages`.

- **Files**: `tools/tree_sitter_languages.py` and `tools/repo_map.py`
- **Change**: Add a public function to `tree_sitter_languages.py`, e.g.:
  ```python
  def extensions_for_languages(requested: set[str]) -> set[str]:
      """Return the set of file extensions for the requested language names."""
      return {ext for ext, (_, lang) in _EXT_TO_LANG.items() if lang in requested}
  ```
  Update `repo_map.py:_filter_by_extension()` to call `extensions_for_languages(requested)` instead of importing `_EXT_TO_LANG` directly. Remove the private import line.
- **Scope**: ~5 lines added to `tree_sitter_languages.py`, 3 lines changed in `repo_map.py`. No behavior change. No test changes needed beyond confirming existing tests pass.
