## Non-Blocking Notes
- [detect_ci.sh:83] Deploy target field misalignment: `"github-actions|||||${deploy_target}|medium"` emits 6 pipes (7 fields), placing deploy_target in the `_lang` slot (field 6) instead of the `DEPLOY_TARGET` slot (field 5). Result: deploy target is silently ignored in `_format_ci_section`'s table display. Functional impact is zero (command extraction ignores this row); purely a display loss in the CI/CD section of PROJECT_INDEX.md.
- [lib/config_defaults.sh] `DETECT_INFRASTRUCTURE_ENABLED` and `DETECT_TEST_FRAMEWORKS_ENABLED` config keys are absent while the other four M12 detections (`DETECT_WORKSPACES_ENABLED`, `DETECT_SERVICES_ENABLED`, `DETECT_CI_ENABLED`, `DOC_QUALITY_ASSESSMENT_ENABLED`) each have one. Infrastructure and test-framework detection always run with no opt-out. The section formatters in detect_report.sh gate on `type -t`, so the omission is functional, but the inconsistency will surprise operators trying to tune a large repo.
- [detect_workspaces.sh:12] `WORKSPACE_ENUM_LIMIT` is honoured as an env var (`${WORKSPACE_ENUM_LIMIT:-50}`) but is not documented in `config_defaults.sh`. Operators looking to tune enumeration on huge monorepos have no in-file hint the knob exists.
- [detect_report.sh:104] `_format_workspace_section` counts subprojects with `grep -c '.'`. When `_format_subprojects` appends `...(N more)`, that marker line is counted as a subproject, inflating the displayed count by 1.
- [init_config.sh:43-46] Model names (`claude-sonnet-4-6`, `claude-opus-4-6`) are hardcoded in the config generator. This is acceptable for a one-time init helper, but should be revisited if Tekhton ever adds a global `PREFERRED_*_MODEL` config tier — today's generated configs would then be inconsistent with it.
