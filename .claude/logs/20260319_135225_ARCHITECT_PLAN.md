## Staleness Fixes

**Fix `_print_box_line` empty-line fallback (`lib/common.sh:64-65`)**

Observation: When `printf` fails, the empty-line branch falls back to a hardcoded
`echo "${_bv}[60 spaces]${_bv}"`, with a comment noting it assumes `_BOX_W=60`.
`printf` failure is extremely rare on bash 4+ systems (project requirement), but
the fallback is fragile ‚Äî if any call site ever passes a different width, the
empty separator will render at the wrong width silently.

Current code (line 64-65):
```
printf '%s%-*s%s\n' "$_bv" "$_bw" "" "$_bv" 2>/dev/null || \
    echo "${_bv}                                                            ${_bv}"  # ~60 spaces
```

Fix: Replace the hardcoded fallback with a width-portable one. The existing
`_build_box_hline` helper already builds a repeated-character string of
arbitrary width ‚Äî reuse it with a space character:
```
printf '%s%-*s%s\n' "$_bv" "$_bw" "" "$_bv" 2>/dev/null || \
    echo "${_bv}$(_build_box_hline "$_bw" " ")${_bv}"
```
This eliminates the hidden `_BOX_W=60` assumption and the misleading comment.
The content branch fallback on line 62 (`echo "${_bv}  ${_content}  ${_bv}"`) is
acceptable ‚Äî it lacks fixed-width padding but renders legibly with actual content.
Leave it unchanged.

**Scope:** `lib/common.sh` only. One-line change. No other files touched.
