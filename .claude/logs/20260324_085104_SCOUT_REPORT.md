# Scout Report: CLARIFICATIONS.md File Structure Bug

## Relevant Files

- `lib/clarify.sh` — The clarification protocol that handles detecting and prompting for answers; contains `handle_clarifications()` which reads from /dev/tty or /dev/stdin
- `lib/intake_verdict_handlers.sh` — The intake verdict handler; contains `_intake_handle_needs_clarity()` which writes questions to CLARIFICATIONS.md and calls handle_clarifications()
- `stages/intake.sh` — The intake stage that runs the PM agent and routes verdicts; calls the verdict handlers
- `stages/coder.sh` — The coder stage that also detects and handles clarifications from CODER_SUMMARY.md
- `prompts/intake_scan.prompt.md` — The PM agent prompt that instructs the agent to produce a "## Questions" section in INTAKE_REPORT.md
- `lib/intake_helpers.sh` — Contains `_intake_parse_questions()` which extracts the Questions section from INTAKE_REPORT.md

## Key Symbols

- `handle_clarifications()` — lib/clarify.sh:79 — Prompts user for answers to blocking questions; writes to CLARIFICATIONS.md
- `_intake_handle_needs_clarity()` — lib/intake_verdict_handlers.sh:130 — Routes NEEDS_CLARITY verdict; writes questions to file then calls handler
- `detect_clarifications()` — lib/clarify.sh:24 — Parses "## Clarification Required" from agent report
- `_intake_parse_questions()` — lib/intake_helpers.sh:88 — Extracts Questions section from INTAKE_REPORT.md
- `run_stage_intake()` — stages/intake.sh:33 — Intake stage entry point; runs PM agent and routes verdict

## Suspected Root Cause Areas

1. **File pre-population problem** — `_intake_handle_needs_clarity()` writes the raw questions section to CLARIFICATIONS.md BEFORE calling `handle_clarifications()`. When `handle_clarifications()` runs, it writes its own header and appends answers. This creates a file with duplicate header structure and may cause confusion in downstream parsing.

2. **Non-interactive mode handling** — In `--complete` mode without a TTY, `/dev/tty` doesn't exist. `handle_clarifications()` at line 112-114 of clarify.sh falls back to `/dev/stdin`. However, stdin may be EOF (no user input available), causing all reads at line 142 to fail and default to `answer="skip"`. This results in all questions being marked `(skipped)` in CLARIFICATIONS.md, not "nonsensical" answers per se, but possibly confusing the user.

3. **Markdown formatting inconsistency** — `_intake_handle_needs_clarity()` writes questions as-is (raw markdown from intake report), which may have list formatting (`- Question text`). But `handle_clarifications()` then writes the same questions with a different markdown format (`## Q: Question text`). This can create structural issues in CLARIFICATIONS.md.

4. **Missing non-blocking handling from intake** — `_intake_handle_needs_clarity()` only handles blocking items written to blocking.txt. It creates an empty nonblocking.txt at line 154, ignoring any NON_BLOCKING items that may exist in the intake agent's output. This means non-blocking clarifications from intake are silently dropped.
