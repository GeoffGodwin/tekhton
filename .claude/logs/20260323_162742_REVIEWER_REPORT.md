## Non-Blocking Notes
- `lib/causality.sh` is 475 lines, well over the 300-line soft ceiling. The file is logically well-organized into sections (init, emit, ID management, cap enforcement, query functions, archival), but `trace_cause_chain`, `trace_effect_chain`, and the query layer could be extracted to a `causality_query.sh` module in a future cleanup pass.
- `_hook_causal_log_finalize` is registered 4th in the finalization sequence (after metrics, before mark_done/archive_reports/emit_run_summary). This means `archive_causal_log()` runs before `mark_milestone_done()` emits `milestone_advance` events. Those events land in the live `CAUSAL_LOG.jsonl` but not in the per-run archive. Data is not lost (the live log accumulates all runs), but the per-run archive snapshot is incomplete for finalization-phase events. No fix needed now — log for future archival refactor.
- `trace_effect_chain` uses `grep -F "\"$current\""` to find events that reference `current`. This is a broad string match — it will match any JSON field containing the event ID string, not just `caused_by`. An event with `"detail":"error near coder.001"` would false-positive as an effect of `coder.001`. No impact on correctness for the primary use case (diagnostic exploration), but worth documenting the limitation.
- `dashboard.sh` sources `dashboard_parsers.sh` using `"$(dirname "${BASH_SOURCE[0]}")/..."` rather than the project-standard `"${TEKHTON_HOME}/lib/..."` pattern. Both work; the inconsistency is minor.
- `_STAGE_BUDGET[intake]` is never set in `tekhton.sh`. All other stages have an explicit budget assignment; intake shows `0` in the dashboard. Consider adding `_STAGE_BUDGET[intake]="${INTAKE_MAX_TURNS:-10}"` after the intake stage block.
- The dashboard generates `.claude/dashboard/data/*.js` files on every run. The hook comment in `hooks.sh` notes these are not archived, but neither `templates/pipeline.conf.example` nor any `.gitignore` template is updated to exclude `.claude/dashboard/`. Downstream projects could accidentally commit these generated files. Add `.claude/dashboard/` to the example `.gitignore` patterns referenced in `_check_gitignore_safety()`.
- `tests/test_finalize_run.sh` header comment still says "11 hooks in deterministic sequence" (line 8) despite the 12th hook (`_hook_causal_log_finalize`) now being registered. The test logic was updated but the comment was not.
- Hook implementation letter labeling in `finalize.sh` skips `k` (jumps from `j` to `l`). The new hook is defined as `# l.` but registered 4th (between `c` and `d`). The letters are decorative and don't affect behavior, but the skip and out-of-order registration make the labels misleading.

## Coverage Gaps
- `trace_effect_chain` has no test for the false-positive-via-detail-field case. Documenting the known limitation in a comment or a skipped test would prevent future confusion.
- No test covers the scenario where `init_causal_log()` is called on a resumed run (non-empty log already exists) — only fresh-run initialization is tested.
- `emit_dashboard_run_state`'s `waiting_for` null-patching (the string replacement `${json/\"waiting_for\":,/\"waiting_for\":null,}`) has no test case for the empty-waiting branch.

## Drift Observations
- `dashboard.sh:source "$(dirname "${BASH_SOURCE[0]}")/dashboard_parsers.sh"` — the only file in the codebase that sources a sibling using `BASH_SOURCE`-relative path instead of `${TEKHTON_HOME}/lib/`. May cause confusion during future refactors.
- `causality.sh:_json_escape` and `dashboard_parsers.sh:_to_js_string/_write_js_file` are both low-level output utilities that live in different modules. As the dashboard grows, having `_json_escape` in causality.sh while output helpers live in dashboard_parsers.sh may cause confusion about where to put new utilities.
- `finalize.sh` hook label sequence (a,b,c,d,e,f,g,h,i,j,**l** — skipping k) and the new hook registered out of alphabetical position relative to its label. Minor doc drift that accumulates if more hooks are added without renaming.
