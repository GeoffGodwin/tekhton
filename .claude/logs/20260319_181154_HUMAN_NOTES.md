# Human Notes

Add your observations below as unchecked items. The pipeline will inject
unchecked items into the next coder run and archive them when done.

Use `- [ ]` for new notes. Use `- [x]` to mark items you want to defer/skip.

Prefix each note with a priority tag so the pipeline can scope runs correctly:
- `[BUG]` — something is broken, needs fixing before new features
- `[FEAT]` — new mechanic or system, architectural work
- `[POLISH]` — visual/UX improvement, no logic changes


## Features
- [ ] [FEAT] We need a configurable usage threshold where we can set Tekhton to not start its next run if the current session usage (claude /usage) passes that threshold. That way if I set it to 90% per 5 hour window it will pause before running the next run when it hits 90% of my session usage.
- [ ] [FEAT] Currently the final commit message is incredible light on details. It would be nice if it included a summary of the changes made, maybe a list of files changed, and perhaps even a brief explanation of why the changes were made (if that can be inferred from the coder's notes or the commit diff). This would make it easier to understand the context of the changes when looking at the commit history later on. It should also auto-commit to the working branch it's on instead of waiting for human input. This wait causes the work to halt which could otherwise continue on to later milestones or tasks.

## Bugs
- [x] [BUG] The current Architecture notes look like nonsense. The method for writing the observations must be broken because it's a mess. Remedy the problem and determine why it's no longer writing neat, orderly individual observations following the agreed upon format. The notes should be clear and actionable, not a jumbled mess.
- [x] [BUG] The --auto-advance feature is not currently working. I recently ran it on Milestone 15.1 and it only completed one then asked me to commit. It should have continued on to the next milestone automatically. This is a critical feature for streamlining the workflow and reducing manual intervention, so it needs to be fixed.
- [ ] [BUG] `resolve_human_notes()` resets [~] → [ ] when CODER_SUMMARY.md is missing at resolution time, even if the pipeline succeeded and features were actually implemented. The final fallback in `lib/notes.sh` should check the pipeline exit code: if the run succeeded (exit 0), treat missing CODER_SUMMARY.md like the COMPLETE path and mark [~] → [x] instead of resetting. Currently features get implemented and committed but HUMAN_NOTES shows them as undone.
- [ ] [BUG] `clear_milestone_state()` in `lib/milestone_ops.sh` exists but is never called anywhere. After a successful milestone completion, MILESTONE_STATE.md persists with stale data from the prior run. This causes subsequent non-milestone tasks (e.g., "Implement HUMAN_NOTES items") to pick up the stale milestone context. `finalize_run()` should call `clear_milestone_state()` after successful milestone archival.
