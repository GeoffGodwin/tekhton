### SF-1: `stages/tester.sh:101` — Self-referential `tee -a "$LOG_FILE"`

**File:** `stages/tester.sh`, line 101

**Current code:**
```bash
grep "Compilation failed for testPath=" "$LOG_FILE" | sed 's/.*testPath=/  /' | sed 's/:.*//' | sort -u | tee -a "$LOG_FILE"
```

**Problem:** `grep` reads `$LOG_FILE` while `tee -a` appends to the same file.
The matched lines are written back into the file that grep is reading, producing
duplicate log entries on every failed compilation.

**Fix:** Capture the deduplicated path list into a variable, print it to stdout
(for the user), and remove the `tee -a "$LOG_FILE"`. The `run_agent` process
already writes the log; this postprocessing step should only display to the user,
not duplicate into the log.

```bash
_failed_paths=$(grep "Compilation failed for testPath=" "$LOG_FILE" \
    | sed 's/.*testPath=/  /' | sed 's/:.*//' | sort -u)
echo "$_failed_paths"
```

The second `grep` on line 104 (which populates `FAILED_FILES`) reads the same
file independently and is unaffected — leave it as-is.

### SF-2: `lib/agent_monitor.sh:211` — Missing comment on inner kill path

**File:** `lib/agent_monitor.sh`, lines 209–214

**Current code:**
```bash
echo "[tekhton] ACTIVITY TIMEOUT — no output or file changes for ${_idle}s. Killing agent." >&3
echo "ACTIVITY_TIMEOUT" > "$_exit_file"
kill "$_TEKHTON_AGENT_PID" 2>/dev/null || true
sleep 2
kill -9 "$_TEKHTON_AGENT_PID" 2>/dev/null || true
_kill_agent_windows
```

**Problem:** The outer `_run_agent_abort` trap also calls `kill "$_TEKHTON_AGENT_PID"`.
A future reader seeing these two kill paths may assume one is redundant and remove
the inner one, breaking activity-timeout enforcement. There is no comment explaining
the isolation.

**Fix:** Add a one-line comment before the `kill` at line 211:
```bash
# This subshell cannot reach the outer _run_agent_abort trap — kill directly.
kill "$_TEKHTON_AGENT_PID" 2>/dev/null || true
```

No logic change. Comment only.

### SF-3: `lib/common.sh:64-65` — `_print_box_line` empty-line fallback missing right border

**File:** `lib/common.sh`, lines 63–65

**Current code (empty-line branch):**
```bash
printf '%s%-*s%s\n' "$_bv" "$_bw" "" "$_bv" 2>/dev/null || \
    echo "${_bv}"
```

**Problem:** When `printf` is absent, the fallback `echo "${_bv}"` prints only
the left border character with no padding and no right border. This produces a
visually broken empty row in the box. The content-line fallback at lines 61–62
has the same issue (`echo "${_bv}  ${_content}"` has no right border).

**Fix:** Update both fallback branches to at least include the right border:
```bash
# Empty line branch:
printf '%s%-*s%s\n' "$_bv" "$_bw" "" "$_bv" 2>/dev/null || \
    echo "${_bv}                                                            ${_bv}"

# Content line branch:
printf '%s  %-*s%s\n' "$_bv" "$((_bw - 2))" "$_content" "$_bv" 2>/dev/null || \
    echo "${_bv}  ${_content}  ${_bv}"
```

The fallback path is only reached when `printf` itself is absent — an extremely
rare condition. The fix keeps the fallback visually consistent without adding
complexity.

## Simplification

Route to **sr coder**.

### SIM-1: `lib/common.sh` — Extract duplicated box initialization block

**Files:** `lib/common.sh`, lines 82–89 (`report_error`) and lines 124–131 (`report_retry`)

**Problem:** Both functions contain identical 8-line blocks that initialize box-drawing
character variables and call `_build_box_hline`. The only difference is the function
name. If one is modified — for example, to change the box width or add a new border
character — the other silently diverges.

**Fix:** Extract a new helper `_setup_box_chars()` that sets script-level (non-local)
variables `_BOX_TL`, `_BOX_TR`, `_BOX_BL`, `_BOX_BR`, `_BOX_H`, `_BOX_V`, `_BOX_W`,
and `_HLINE`. Both `report_error` and `report_retry` call `_setup_box_chars 60` and
then reference these variables directly.

Since the variables are script-level (no `local`), they are visible to both callers
without requiring bash array-passing tricks. The names use `_BOX_` prefix (uppercase)
to distinguish from the current `_box_` local names and avoid collision with any
other script globals.

Scope: modify only `lib/common.sh`. No other files reference these internals.

### SIM-2: `lib/config.sh` — Extract defaults block to `lib/config_defaults.sh`

**File:** `lib/config.sh` (342 lines — exceeds the 300-line ceiling)

**Problem:** `lib/config.sh` is 342 lines, 42 over the non-negotiable 300-line
ceiling. The file grew as each milestone added new `MAX_*`, `ENABLED`, and
`TRANSIENT_RETRY_*` defaults. The defaults block is a natural extraction boundary —
it is purely declarative and has no dependency on the `_parse_config_file()` or
`load_config()` logic.

**Fix:** Create `lib/config_defaults.sh`. Move the block of `: "${KEY:=value}"`
and `_clamp_config_value` calls into it. `lib/config.sh` sources
`lib/config_defaults.sh` at the bottom of `load_config()`, after the file parse
and validation, so all project-supplied values take precedence before defaults
are applied.

**Constraints:**
- `config_defaults.sh` must begin with `#!/usr/bin/env bash` and `set -euo pipefail`
- `tekhton.sh` does NOT need to source it separately — only `config.sh` does
- `ARCHITECTURE.md` "File Ownership" table should be updated to list the new file
- All existing tests must pass

Estimate: ~80–100 lines extracted, leaving `config.sh` at ~245 lines.

### SIM-3: `lib/agent_monitor.sh` — Extract file-change and reset helpers

**File:** `lib/agent_monitor.sh` (400 lines — exceeds the 300-line ceiling)

**Problem:** `lib/agent_monitor.sh` is 400 lines, 100 over the ceiling. The file
mixes two distinct responsibilities: (1) FIFO-based invocation and real-time
monitoring (`_invoke_and_monitor`, Windows detection, the abort trap), and (2)
post-invocation helpers (`_detect_file_changes`, `_count_changed_files_since`,
`_reset_monitoring_state`). The helpers at lines 329–400 have no callers inside
`_invoke_and_monitor` — they are called by `agent_helpers.sh` and `agent_retry.sh`.

**Fix:** Create `lib/agent_monitor_helpers.sh`. Move `_detect_file_changes()`,
`_count_changed_files_since()`, and `_reset_monitoring_state()` into it.
`agent.sh` sources `agent_monitor_helpers.sh` alongside `agent_monitor.sh`.

**Constraints:**
- `_kill_agent_windows` and `_TEKHTON_AGENT_PID` remain in `agent_monitor.sh`
  because `_reset_monitoring_state` calls them. Either keep `_reset_monitoring_state`
  in `agent_monitor.sh` (simplest: only move `_detect_file_changes` and
  `_count_changed_files_since`) or ensure `agent_monitor.sh` is sourced before
  `agent_monitor_helpers.sh` so the dependency is available.
- Simplest correct split: move only `_detect_file_changes()` and
  `_count_changed_files_since()` to `agent_monitor_helpers.sh` (~35 lines).
  This brings `agent_monitor.sh` to ~365 lines — still over. Alternatively,
  move all three helpers (~75 lines) and source `agent_monitor.sh` first from
  `agent.sh` so `_kill_agent_windows` is defined when `_reset_monitoring_state`
  loads. This brings `agent_monitor.sh` to ~325 lines — still slightly over.

  **Recommended approach:** Move all three helpers AND the `_TIMEOUT_KILL_AFTER_FLAG`
  setup block (lines 10–13, ~4 lines) and the Windows detection block (lines 16–30,
  ~15 lines, already self-contained) to a new `lib/agent_env.sh`. Leave only
  `_invoke_and_monitor()` and the FIFO reader subshell in `agent_monitor.sh`.
  Aim: `agent_monitor.sh` < 280 lines; `agent_env.sh` ~120 lines.

  **Simpler acceptable split:** if the above is too broad, extract only
  `_detect_file_changes()` and `_count_changed_files_since()` to
  `agent_monitor_helpers.sh` for now and accept that agent_monitor.sh
  at ~365 lines is an improvement. Document the remaining overage in a
  code comment for the next audit pass.

- `ARCHITECTURE.md` must be updated to list any new lib file
- All existing tests must pass; `shellcheck` and `bash -n` must pass
