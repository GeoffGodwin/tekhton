# Scout Report: Milestone 21 — Version Migration Framework & Project Upgrade

## Summary
Milestone 21 implements a general-purpose project configuration migration system that automatically upgrades project configs when Tekhton versions advance. The system is non-destructive and idempotent, featuring version watermarking, smart version inference for legacy projects, backup/rollback support, and integration with startup, initialization, and diagnostic flows.

## Relevant Files

- `lib/migrate.sh` — Core migration framework with version detection, migration runner, backup/rollback, and startup integration
- `migrations/001_to_002.sh` — V1→V2 migration (add context budget keys, v2 config keys, agent role files)
- `migrations/002_to_003.sh` — V2→V3 migration (add v3 config keys, security/intake agents, milestone DAG, dashboard, section headers)
- `migrations/README.md` — Migration authoring guide documenting the interface contract for future versions

## Key Symbols

### Core Migration Functions (lib/migrate.sh)
- `detect_config_version()` — Infers V1/V2/V3 from artifact inspection (MANIFEST.cfg, CONTEXT_BUDGET_PCT, SECURITY_AGENT_ENABLED)
- `run_migrations()` — Loads all migration scripts, filters by version range, runs in order, stops on failure
- `backup_project_config()` — Creates .claude/migration-backups/pre-X.Y-to-X.Z/ with pipeline.conf, CLAUDE.md, agents/, MANIFEST.cfg
- `rollback_migration()` — Restores from backup directory, updates TEKHTON_CONFIG_VERSION
- `check_project_version()` — Compares config version vs running version, prompts/auto-applies, handles --complete mode

### Migration Script Interface (in each migrations/*.sh)
- `migration_version()` — Returns "X.Y" version this migration targets
- `migration_check()` — Returns 0 if migration needed, 1 if already applied (idempotency check)
- `migration_apply()` — Performs migration, returns 0 on success or non-zero on failure (stops chain)
- `migration_description()` — Returns human-readable description of what migration does

### Config Version Inference Heuristics (lib/migrate.sh)
- Presence of MANIFEST.cfg + .claude/milestones/ → V3.0
- CONTEXT_BUDGET_PCT in pipeline.conf → V2.0
- pipeline.conf exists with only basic keys → V1.0
- .claude/ exists but no pipeline.conf → V0.0 (pre-Tekhton, prompt --init)

### Milestone Validation (during load in lib/milestones.sh or lib/milestone_dag.sh)
- `validate_milestone_migration_section()` — Checks each milestone .md for Migration impact: section
- Warning logged if missing, added to Watchtower health, noted in PM agent (M10) intake report

## Suspected Root Cause Areas

1. **Version inference heuristic accuracy** — Must conservatively distinguish V1 vs V2 vs V3 from incomplete artifact sets. Uses presence of MANIFEST.cfg, specific config keys, and directory structure. See lib/migrate.sh `detect_config_version()`.

2. **Non-destructive guarantees in migrations** — Both 001_to_002.sh and 002_to_003.sh must never delete user files or overwrite customized agent roles. Verify: append-only for config keys, check-before-copy for agent files, backup exists before any changes. See migration scripts.

3. **Idempotency and failed-state recovery** — Each migration's `check()` function must correctly detect already-applied state. If migration fails mid-way, re-running must be safe (no duplicate config keys, no failed directory creation, etc.). See migration `check()` functions.

4. **Config file parsing and appending** — pipeline.conf is sourced as bash, not a standard format. Adding new keys safely requires careful sed/awk or append-only logic. Modifying existing values is risky. See lib/migrate.sh config update logic and 001_to_002.sh, 002_to_003.sh.

5. **Startup integration flow** — check_project_version() called after config load but before pipeline stage runs. Must handle --complete/--auto-advance (auto-apply), --migrate flag (explicit), and interactive prompt (default). Must not block autonomous runs. See tekhton.sh startup sequence and check_project_version() logic.

6. **Backup directory management** — Backups accumulate in .claude/migration-backups/. Future --migrate --cleanup-backups will prune old backups (keep last 3). See backup_project_config() and rollback_migration() in lib/migrate.sh.

7. **Version comparison and sorting** — V1→V3 requires running both V1→V2 and V2→V3 in sequence. Migration runner must sort by version and filter to applicable range. Must handle X.Y semver format correctly. See run_migrations() in lib/migrate.sh and _semver_lt() helper.

8. **Agent role file updates** — V2→V3 migration copies new agent role files but must NOT overwrite user customizations. Must detect old template versions (check for version marker comment) and preserve user changes. See 002_to_003.sh agent role copy logic.
