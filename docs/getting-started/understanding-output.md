# Understanding Output

After a pipeline run, Tekhton produces several report files. Here's what each one
contains and how to read it.

## Report Files

### CODER_SUMMARY.md

The coder agent's self-report. Contains:

- **Status** — `COMPLETE` or `IN PROGRESS`
- **What Was Implemented** — Bullet list of changes made
- **Files Modified** — Every file that was created or changed
- **Architecture Change Proposals** — Any structural changes the coder made that
  deviate from the documented architecture
- **Remaining Work** — What's left if status is `IN PROGRESS`

This is the first file to check after a run. If the status is `IN PROGRESS`, the
coder ran out of turns or hit a blocker.

### REVIEWER_REPORT.md

The code reviewer's assessment. Contains:

- **Verdict** — `APPROVED`, `CHANGES_REQUIRED`, or `REPLAN_REQUIRED`
- **Issues Found** — Categorized by severity (blocker, major, minor)
- **Architecture Change Proposals** — The reviewer's verdict on any ACPs the coder
  proposed (ACCEPTED or REJECTED with rationale)
- **Drift Observations** — Patterns the reviewer noticed that may indicate
  architectural drift

If the verdict is `CHANGES_REQUIRED`, Tekhton automatically routes issues to
a rework agent.

### SECURITY_REPORT.md

The security agent's findings. Contains:

- **Severity-rated findings** — CRITICAL, HIGH, MEDIUM, LOW, INFORMATIONAL
- **Remediation status** — Whether each finding was fixed automatically
- **Unfixable issues** — Items that require human attention (escalated per your
  `SECURITY_UNFIXABLE_POLICY` setting)

### TESTER_REPORT.md

The test agent's output. Contains:

- **Tests written** — List of test files and test cases created
- **Test results** — Pass/fail status for each test
- **Coverage notes** — What the tests cover and any gaps

### INTAKE_REPORT.md

The PM/intake agent's task evaluation. Contains:

- **Clarity score** — How clear and actionable the task description is
- **Scope assessment** — Whether the task is appropriately sized
- **Recommended tweaks** — Suggestions for improving the task description

## RUN_SUMMARY.json

A machine-readable summary of the entire pipeline run. Key fields:

| Field | Description |
|-------|-------------|
| `task` | The task description |
| `status` | Overall pipeline status |
| `stages` | Per-stage results (timing, turns used, verdict) |
| `total_time` | Wall-clock duration |
| `agent_calls` | Number of agent invocations |
| `milestone` | Which milestone was being worked on (if any) |

## HEALTH_REPORT.md

Generated by `tekhton --health`. Contains:

- **Overall score** — 0-100 composite score
- **Category scores** — Tests, quality, dependencies, documentation, hygiene
- **Belt rating** — White through black belt (a quick visual indicator)
- **Recommendations** — Specific improvements to raise your score

## Reading Terminal Output

During a pipeline run, the terminal shows:

- **Stage headers** — Which stage is running (e.g., `[CODER]`, `[REVIEWER]`)
- **Agent status** — Turn count, timing, model used
- **Build gate results** — Pass/fail for compile and lint checks
- **Verdicts** — Review verdict and any rework routing decisions

If a stage fails, the terminal output includes the error classification and
suggested recovery action.
