## Your Task

Generate a complete DESIGN.md that documents this project's current state.
Follow the template structure below, populating each section from codebase
evidence. Every claim must be traceable to something in the project index,
detection report, or sampled files.

## Required Sections

### 1. Project Overview
- Project name and purpose (from README, package manifests, or inferred from code)
- Target audience and use case
- Current state: is this a mature project, early development, prototype?
- Key dependencies and their roles

### 2. Developer Philosophy
- Architectural patterns actually used (observed in code, not aspirational)
- Code organization principles (how are files and modules structured?)
- Dependency management approach
- Error handling patterns observed
- Testing philosophy (test coverage level, testing frameworks used)

### 3. Architecture
- System decomposition: major modules/packages and their responsibilities
- Data flow: how requests/events/data move through the system
- Module boundaries and dependency relationships
- External integrations and APIs consumed
- State management approach (if applicable)

### 4. Directory Structure
- Annotated directory tree showing purpose of each major directory
- Key files and their roles
- Configuration file locations
- Test file organization

### 5. Core Systems
For each major subsystem or module:
- Purpose and responsibility
- Key files and entry points
- Public interfaces / exports
- Dependencies on other subsystems
- Configuration knobs (if any)

### 6. Data Model
- Key data structures, schemas, or models
- Database schema or storage approach (if applicable)
- Data validation patterns
- Serialization/deserialization approach

### 7. Configuration
- Config file format and loading mechanism
- Environment variables used
- Default values and override hierarchy
- Required vs optional configuration

### 8. Build and Deploy
- Build system and commands
- Test commands and frameworks
- Lint/format commands
- Deployment approach (if observable from config)
- CI/CD setup (if observable from config files)

### 9. Conventions and Patterns
- Naming conventions (files, functions, classes, variables)
- Import ordering patterns
- Comment and documentation style
- Git workflow (if observable from branch structure or config)

### 10. Technical Debt and Open Questions
- Inconsistencies observed between different parts of the codebase
- Missing test coverage areas
- Undocumented or unclear subsystems
- TODO/FIXME/HACK comments found in code
- Architecture decisions that appear to have evolved organically without
  clear documentation

## Output Rules

1. **Output DESIGN.md content directly to stdout.** Do NOT use any tools to
   write files. Start with `# [ProjectName] — Design Document`.

2. **Be evidence-based.** Every claim about the codebase must be supported by
   something in the project index or sampled files. Do not invent patterns
   that are not observable.

3. **Distinguish observation from inference.** When you are confident about a
   pattern (multiple files follow it), state it directly. When you are inferring
   from limited evidence, say "appears to" or "likely" and flag it.

4. **Flag uncertainty.** If a section cannot be adequately populated from the
   available evidence, say so explicitly rather than filling it with generic
   content.

5. **Reference file paths.** When describing patterns, reference the actual
   file paths from the project index where those patterns are observed.

6. **Target length: 300–1200 lines** depending on project complexity.
