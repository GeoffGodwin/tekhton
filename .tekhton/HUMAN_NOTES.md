- [x] [BUG] Live TUI sidecar's alternate screen gets kicked back to the main screen repeatedly during the coder stage's "Running completion tests" substage, exposing per-stage `print_run_summary` output and other stdout writes (Run Summary banner, log lines) on the main screen. Symptom looks like "the TUI keeps exiting" — what's actually happening is that `tput rmcup` / `tput cnorm` / `stty icrnl` escape sequences leak to the shared `/dev/tty` from inside the test suite, switching the terminal out of alternate screen mode while the parent's `tools/tui.py` rich.live is still running. Introduced by commit 5876c91 ("TUI sidecar orphaned after build-gate-failure exit") which (a) added a terminal-restoring safety net to `lib/tui.sh:223-225` inside `tui_stop` and (b) added two new tests that call `tui_stop` directly: `tests/test_tui_stop_orphan_recovery.sh` (calls `tui_stop` 4× with various pidfile / `_TUI_ACTIVE` combos) and `tests/test_tui_orphan_lifecycle_integration.sh` (also spawns a real `tools/tui.py` sidecar bound to its own status file, which then renders to the parent's `/dev/tty` alongside the real sidecar). Both tests stub `log` / `warn` / `error` / `success` etc. before sourcing `lib/tui.sh`, but neither stubs `tput` or `stty`, so the safety-net writes hit the parent shell's terminal. Repro: run `tekhton.sh --auto-advance N <ms>` on a project whose `TEST_CMD` runs `tests/run_tests.sh`; during the coder build gate's completion-tests substage, watch the alt-screen flicker every time one of the new tests calls `tui_stop`.
